Data & Use Case Architecture

How Terminal6 turns multi-channel commerce data into agent reasoning

A walk-through of how data is stored, how reusable computation is built on top of it, and how those pieces are combined to answer operator questions.

May 2026 Live agent: Amazon Performance Marketing contact@terminal6.io

1. The problem

A brand selling on Amazon, Shopify, Flipkart, Meta, Google, GA4, and Unicommerce gets data in seven formats. Answering "why did Amazon revenue drop 35% yesterday" today requires pulling and reconciling numbers from each tool, identifying which channel and products moved, and diagnosing the cause across paid, organic, inventory, pricing, and returns. Terminal6 does each of these in code.

The system has three parts:

  1. A database that stores all channels in the same shape.
  2. A set of small Python modules, each computing one signal.
  3. Markdown skill files that describe how to investigate a specific question. An LLM reads them; the system fetches the declared data; the skill writes a structured answer.

2. The database — four layers

A
Raw
Untouched API responses, one row per call. Audit + replay.
B
Canonical
35+ tables, all channels in one shape. Single FK hub (channel_listing_id) so cross-channel queries are one join.
C
Computed
Derived signals materialised as tables (baseline grounding + deviation calculations, P&L per ASIN, keyword cohorts, listing health, etc.).
D
Decisions
Every agent run writes a row: which skill, what input, what output, parent run. Full audit tree.

Layer C — building blocks

Layer C is where the building blocks live. Examples:

  • unit_economics.py → asin_unit_economics (per-ASIN ASP, COGS, CM2, CM3, return rate)
  • keyword_classification.py → keyword_classifications (Branded / Must-Win / Generic / Competitor)
  • creative_classification.py → 7-bucket Meta creative tags
  • listing_health.py → cross-channel listing scores
  • realtime_pnl.py → per-ASIN profit before settlement reconciles

There are 17 such modules. Each is 100–400 lines, has one responsibility, runs daily, writes one table.

Rule: a Layer C table earns its existence if it (a) changes grain, (b) joins two or more Layer B sources, or (c) derives a novel signal. Same-grain derived columns belong on the canonical Layer B table.

3. The reasoning layer — skills

The data layers are deterministic code. The reasoning layer uses an LLM with one principle: the LLM does not decide what data to fetch. A markdown skill file declares its inputs; the system fetches them; the LLM reads the data and writes an answer.

Every skill follows the same structure:

# Skill: <name> ## Trigger — when this skill runs ## Context Required — which knowledge cards to load ## Data Required — exact tables and fields needed (per factor) ## Decision Logic — factors to check, in order ## Actions — what the skill can recommend ## Edge Cases — when to mark inconclusive

Skill files never name a brand. Brand-specific facts live in knowledge cards:

  • Brand card — categories, channels, target ACoS, fulfilment model
  • Category card — how a category behaves (mobile accessories: long-tail, high substitution; kids' lunch: head category, individual SKU tracking matters)
  • Region card — market context (India: salary cycle drives 1st-of-month spike, COD %, return-rate norms)

Skills are organised in 4 levels: L0 brand-head triage, L1 senior managers (marketing, operations, finance, category, channels), L2 channels, L3 function skills under a channel. About 150 skill files today.

5. Skill structure — today and evolution

Today the agent is a single skill file. The runner loads ads_review.md (~3.2K tokens) as system prompt + the 9-input brief as user message. One LLM call. Done.

Tomorrow the monolith splits. Each function becomes its own skill file:

sr_marketing/ ├── triage.md (gate — is anything wrong with marketing today?) ├── investigate.md (cross-channel parent — invokes channel agents) └── ads_review.md (today's monolith — splits into channel agents below) channels/amazon/ppc/ ├── investigate.md ├── keyword_intelligence.md (search terms, cohorts, harvest) ├── structure_audit.md (campaign sprawl, ad-group hygiene) ├── bid_optimization.md (bids constrained by margin + cover) └── creative_audit.md (image quality, listing fixes) channels/meta_ads/ (same pattern) channels/google_ads/ (same pattern)

Sub-skills read different subsets of Layer C

keyword_intelligence → keyword_intelligence_daily + amazon_search_term_harvest + asin_unit_economics structure_audit → campaign_efficiency_daily + amazon_campaign_metrics bid_optimization → campaign_efficiency_daily + asin_unit_economics + asin_forward_cover + keyword_intelligence_daily creative_audit → listing_health_snapshots + product_reviews

Each sub-skill pulls only what its Data Required declares. Token budget per sub-skill stays bounded. The materialised tables compound: asin_unit_economics is read by 3 sub-skills; campaign_efficiency_daily by 4. Building each table once pays for itself across the marketing function — and the same tables serve operations, finance, and category sub-skills downstream.

The skill itself becomes brand-agnostic. The same keyword_intelligence.md works for a kids' lunch brand in India and a pet food brand in the US — the brand card and region card differ; the skill does not. Every new brand onboarded benefits from every skill ever written.

6. Scaling the answer — adding data inputs (horizontal)

Each new input wired in makes the diagnosis sharper. Examples not in the brief today:

  • Competitor price tracking — distinguish "we were undercut" from "creative fatigue"
  • Buy Box win-rate per day — catch ad spend lost to a competitor winning the box
  • Returns by reason per ASIN — fixable (defect, listing) vs. structural (sizing)
  • Customer LTV per first-purchase ASIN — reframe a "high-CAC scaler" as profitable if LTV exceeds payback

Pattern: new compute module → new Layer C table → new row in skill's Data Required. Cost grows linearly with brief size (~100–300 tokens per input). Output structure unchanged.

7. Scaling hierarchically — agents that invoke agents (vertical)

Today the Amazon agent is a leaf. The next build is a parent Senior Marketing agent that runs the four channel leaves (Amazon, Meta, Google, Shopify) in parallel and synthesises their outputs.

The parent's Data Required does not list raw tables — it lists child agent outputs. Each child returns its 6-bucket JSON. The parent reads those four JSONs (~2K tokens each) and reasons cross-channel: which channels are over- or under-funded, where to reallocate, where the same customer is being targeted on multiple channels at once.

A full multi-channel marketing review on Sonnet costs roughly 4 × ₹70 (children) + ₹40 (parent) ≈ ₹320. If the parent runs on Haiku (it's just synthesising compact JSON inputs), the total drops to ~₹285.

The same pattern compounds upward to a daily morning brief:

L0 Brand Head — produces the morning brief ├── L1 Senior Marketing │ ├── Amazon (running today) │ ├── Meta (data flowing, agent partial) │ ├── Google (data flowing, agent partial) │ └── Shopify (data flowing, agent stub) ├── L1 Senior Operations (Inventory, Returns, Fulfilment) ├── L1 Senior Finance (P&L health, Working capital) ├── L1 Senior Category └── L1 Senior Channels

A full daily brief is a tree of 20–30 agent invocations, mixing Haiku (triage gates, narrow function skills) and Sonnet (investigations). L0 triage decides which branches fire — calm day ~₹30 (mostly Haiku triage), active day ~₹500 (Sonnet investigations across multiple branches). Cost tracks situation complexity, not batch size.

8. Brand-tuning — universal rules + brand-specific inputs

A skill is written once and runs for every brand — but every brand has different specifics. Basil's brand terms are basil / yourbasil / bento; its competitors are Milton, Cello, Rabitat; its excluded category is cookware. A mobile-accessories brand has a completely different lexicon.

The pattern: universal logic stays in code or skill markdown; brand-specific inputs live in JSON next to the brand card.

Live example — the keyword classifier

core/data/compute/keyword_classification.py runs weekly and tags every keyword for every brand into 6 cohorts. The rule structure is universal — it does not change per brand:

1. Branded — keyword matches brand_terms regex 2. Competitor — matches a competitor_terms group 3. Excluded — matches global ∪ brand-level excluded terms 4. Generic — search volume above head_term threshold 5. Must-Win — meets share / clicks / volume thresholds 6. Other — default

What changes per brand:

  • Brand lexicon — ["basil", "yourbasil", "bento"] vs. ["spigen", "ringke"]
  • Competitor lexicon — {"value_negate": ["milton", "cello"], "premium_comparative": ["rabitat", "yumbox"]}
  • Excluded terms — kitchenware out for kids' lunch boxes; smartphones out for phone cases
  • Thresholds — head-term cut-off 50K/month for kitchenware, ~10K for niche phone cases

These live in knowledge/brand_cards/{brand}/classification.json next to the brand's prose card. At runtime the loader merges global defaults with brand overrides:

cfg = deep_merge(_defaults.json, brand/classification.json) classifier.run(cfg)

Onboarding a new brand

3 DB inserts (BrandProfile, brand_users, oauth_tokens per channel) + 1 brand card + 1 classification.json. The JSON config is built in two steps:

  1. LLM bootstrap — at onboarding, an LLM reads the brand's prose card + top 200 search keywords + product titles, and emits a classification.json.proposal with lexicons.
  2. Operator review — T6 ops diffs the proposal, edits where wrong, renames to .json, commits.

The bootstrap LLM call happens only at onboarding, new-category SKU launches, and quarterly reviews — not at every classification run. Tested end-to-end with Dish Dash — zero to running daily in under a day.

How brands compound — promotion to globals

A quarterly script proposes updates to _defaults.json:

  • A lexicon item promotes if used by ≥3 brands AND not contradicted by any
  • A threshold promotes if the median across brands is stable for 2 quarters AND ≥70% of brands sit within 20% of the median

Never auto-merges — emits a PR diff for review.

The first kitchenware brand had to specify "exclude pressure cooker, mixer, kadai." Once 3 kitchenware brands all exclude these, the items move to global defaults — and the 4th kitchenware brand gets them for free.

9. Why this matters

  1. Predictable cost. Token budgets bounded by skill + brief size, both declared. Per-brand-per-day cost can be quoted in advance.
  2. Auditable reasoning. Every recommendation traces to a tree of rows in decision_records, each linked to the data it received.
  3. Reusable pieces. 17 compute modules + ~150 skill files apply to every brand, every channel, every question that fits the pattern.
  4. Two scaling axes that compound. Agents get sharper as more data is wired in (horizontal). Agents become building blocks for larger agents above them (hierarchical).
  5. Brand-tuned at scale. Universal logic in code; brand specifics in JSON. Items used by ≥3 brands promote into global defaults. Each new brand makes the next brand cheaper to onboard.
